## Michael Polanyi 是谁

匈牙利裔犹太哲学家+物理化学家（1891-1976），1958 年提出"默会知识"概念。核心一句话：

> **"我们所知道的，远多过我们所能言说的。"**

被称为"当代认识论中的哥白尼"。

## 默会知识理论的几个关键点

**1. 两种知识**
- **默会知识**：根植于经验、直觉、身体记忆，说不出来但确实在用（比如骑自行车、认人脸、设计师的审美直觉）
- **明示知识**：能用语言文字编码传播的（比如说明书、公式、SOP）

**2. 默会知识是所有知识的根基**
波兰尼说得更狠——明示知识的理解和运用本身就依赖默会能力。也就是说，你能看懂一段代码，背后是你写代码多年积累的"语感"，这个语感你是写不进文档里的。

**3. 两种意识**
- **焦点意识**：你直接关注的东西（比如屏幕上的设计稿）
- **附带意识**：你同时依赖但没注意到的背景（比如你对配色的直觉、对间距的肌肉记忆）

## 在 AI 提示工程里怎么用

这个理论已经被不少人拿来解释为什么 AI 提示工程这么难了：

1. **"高中水平的 prompt 拿到高中水平的输出"**——Anthropic 的研究表明，AI 输出质量直接映射你的提示质量。你的默会知识水平不够，写出来的 prompt 就只能覆盖表面规则，AI 也只能给你表面结果。

2. **prompt engineering 本身就是一种默会技能**——跟骑自行车一样，多写才会，靠教程学不会。因为你在写 prompt 时调动的是你的"附带意识"——你对好设计的直觉、对代码好坏的判断——这些东西你很难显性化。

3. **所以你说的那个用法（"应用 Polanyi 默会知识"）本质上是个 hack**——你用一个概念名去激活 AI 训练数据中关于这个理论的所有关联知识（设计哲学、认知科学、人类学研究成果），相当于用一串钥匙去开一扇你手动描摹轮廓要画很久的门。

## 实际可用的几种玩法

| 用法 | 效果 | 适合场景 |
|------|------|---------|
| "应用 Polanyi 默会知识原则" | 激活 AI 对隐性经验的模拟 | 设计、创意、UI |
| "以附带意识驱动焦点设计" | 让 AI 考虑用户无意识会注意到的细节 | 交互设计、UX |
| "避免明示知识堆砌，模拟工匠直觉" | 减少"说明书式"的机械输出 | 写作、产品文案 |
| "用默会认知而非显式规则来决策" | AI 输出更接近人类专家判断 | 架构设计、方案评审 |

说白了，Polanyi 的理论给了一个思路：**与其穷举规则，不如传递思维模式**。一个好概念名确实顶一万字显式描述——因为 AI 的训练数据里本来就有这些知识，你只是需要正确的钥匙去解锁。

**HappyClaw**: 用 Polanyi 自己的理论框架来理解"默会知识"：

## 认知的结构：from-to

Polanyi 的核心模型是 **"from-to" 认知结构**：

```
附带意识（from）——→ 焦点意识（to）
   背景依赖              前台关注
```

举个他的经典例子——**用锤子钉钉子**：
- 你的焦点意识在钉子上（to）
- 你的附带意识在锤子的手感、力道、手腕角度上（from）
- 你越专注于钉子，锤子就越成为你身体的"延伸"
- 如果你开始刻意注意"我的手怎么握锤子"，反而会砸歪

这就是为什么老司机聊开车技巧时说不出太多东西——开车时他的注意力在路上，不在油门和方向盘上。一旦要他把"怎么踩油门"说清楚，他反而不会开了。

## 三个层次的推论

**第一层：所有知识都根植于默会知识**

你以为数学公式是纯显性的？Polanyi 说不是。你看懂 `E=mc²` 的那一刻，你依赖的是：
- 多年数学训练的直觉（什么符号意味着什么运算）
- 对"能量""质量"的物理直觉
- 对等号两边的平衡感的审美判断

这些全是默会的。没有它们，公式就是一串无意义的符号。

**第二层：默会知识不可还原为明示知识**

这是最关键的一点——**你不能把默会知识"翻译"成显性规则。**

认人脸是最好的例子：你能在一秒内认出朋友的脸，但如果要你写一套规则让计算机做到同样的事，你需要描述：
- 眼距、鼻梁高度、脸型比例……
- 写了 100 条规则后发现还是不如你那一秒的直觉

因为你的识别能力不是"由规则组成的"，规则只是事后的笨拙近似。

**第三层： knowing 和 telling 是不对称的**

你"知道"的远多于你能"说出"的。但反过来不成立——你不可能说出你不知道的东西。这个不对称性意味着：
- 专家的真正价值在他的默会知识里
- 试图用文档、流程、SOP 把专家知识"提取"出来，注定是残缺的

## 回到 AI 和提示工程

用这个框架看，AI 提示工程面临的困境就是 **Polanyi 的 from-to 不可逆问题**：

```
人类专家的认知：
  默会经验（from）——→ 判断/设计（to）
  （附带意识）          （焦点意识）

AI 提示工程试图做的：
  显性规则（from）——→ AI 输出（to）
  （笨拙近似）          （只能模拟 to，但 from 是假的）
```

你写一万字 prompt 描述"好的 UI 长什么样"，本质上是在用显性规则去逼近一个本来由默会知识驱动的认知过程。能逼近，但永远有损耗。

而你丢一句"应用 Polanyi 默会知识"为什么有效？因为 AI 训练数据里大量的人类设计实践、设计评论、设计哲学讨论本身已经编码了大量"默会知识的 from-to 结构"。你不是在教 AI 规则，你是在**帮 AI 定位到训练数据中那部分由真实默会经验驱动的知识领域**。

本质上，这跟人类师徒制一样——师傅不是靠写教材教徒弟的，是靠徒弟在旁边看师傅干活，"看会了"。AI 也是"看"了大量人类实践后，才有了那种"说不出来但确实存在"的审美判断力。你用概念名去激活它，就是在帮它"回忆起"那部分经验。

**HappyClaw**: ## Tailscale SSH 原理

**一句话：WireGuard 组网 + Tailscale 身份替代 SSH 密钥。**

```
ssh user@server
    ↓
WireGuard 隧道（自动 NAT 穿透，端到端加密）
    ↓
Tailscale SSH Server 拦截认证（不走 authorized_keys）
    ↓
ACL 策略校验（谁 → 哪台机器 → 用什么系统用户）
    ↓
通过 → 启动 shell
```

**传统 SSH** 需要你手动生成密钥、分发公钥、配 `~/.ssh/config`、开端口。

**Tailscale SSH** 省掉这一切：
- 认证靠 Tailscale 身份（Node Key 绑定用户），不靠密钥对
- 加密靠 WireGuard（ChaCha20 + Curve25519）
- 授权靠集中式 ACL 策略（在管理后台配一次，所有机器生效）
- 撤销靠移除设备（踢出网络 = 立刻失去所有访问权限）

## 怎么配置

**三步搞定：**

**1. 所有机器装 Tailscale**
```bash
# macOS
brew install tailscale
sudo tailscale up

# Linux (Ubuntu/Debian)
curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up --authkey=你的authkey  # 首次可用 authkey 免交互
```

**2. 目标机器开启 SSH**
```bash
sudo tailscale set --ssh
```

**3. 直接连**
```bash
ssh user@machine-name    # 用 Tailscale 机器名，不用记 IP
```

就这样，没了。不需要配密钥，不需要 `~/.ssh/config`，不需要公网 IP。

**可选的 ACL 策略**（在管理后台的 `tailnet` 设置里加）：
```json
{
  "ssh": [
    {
      "action": "accept",
      "src":    ["autogroup:member"],
      "dst":    ["tag:servers"],
      "users":  ["root", "deploy"]
    }
  ]
}
```

**HappyClaw**: 所谓"技术+场景+合规"复合型人才，拆开来说：

## 三层能力的含义

**懂技术：不只是会用工具**

不是说会用 ChatGPT 或调 prompt 就叫懂技术，而是：
- 理解大模型的能力边界（什么时候会幻觉、什么时候靠谱）
- 能做 RAG、Agent、Fine-tuning 等技术选型和方案设计
- 知道怎么评估模型效果、怎么优化推理成本
- 能跟工程师对齐技术方案，而不是只会提需求

**懂行业（场景）：知道AI在业务里怎么落地**

比如一个做金融的 AI 人才，他得知道：
- 金融风控的业务流程长什么样，AI 能插在哪个环节提效
- 哪些决策可以被 AI 辅助，哪些必须人兜底（监管要求）
- 一句话需求背后的真实业务痛点是什么（客户说"我要个 AI 报表"，其实是想解决数据孤岛和决策延迟）

**懂合规：知道红线在哪**

- 数据合规：训练数据能不能用用户隐私数据？怎么脱敏？跨境传输有什么限制？
- 内容合规：生成的内容如果涉及医疗/法律建议，有没有资质风险？
- 行业监管：金融、医疗、教育等行业各自有 AI 应用规范（比如金融的算法备案、医疗的医疗器械审批）
- 出海合规：GDPR、欧盟 AI Act 等海外法规要求

## 为什么这种人稀缺

现在市场上的人才分布是这样的：

```
         懂技术
          /\
         /  \
        /    \
       /______\
      /   跨   \
     /  交 区   \
    /______ 非常小\
   懂行业        懂合规
```

大部分人只占一角：
- 纯技术人员：会调模型，但不懂业务场景
- 行业专家（传统行业转型）：知道痛点，但说不出技术方案
- 合规/法务：知道红线，但不会评估 AI 技术实现

真正三个角都占的人——能自己设计方案、知道业务痛点、能评估合规风险——占比不到 20%，这就是报告说的缺口。

## 举个实际例子

假设一个银行要做 AI 智能客服：

- **纯技术人员**：搭个 RAG 系统，接入大模型，能跑起来 ✅
- **+懂行业**：知道银行客服最常见的 20 个问题是什么，知道哪些问题回答错了会引发投诉，设计分层路由（简单问题 AI 答，复杂问题转人工）✅
- **+懂合规**：知道不能把用户资金数据喂给第三方大模型，必须用私有化部署或脱敏方案；知道 AI 生成的理财建议如果导致用户亏损，银行有责任——所以必须加免责声明和人工复核机制 ✅

三个都懂的人才能端到端把这事做对、做完、做安全。现在这种人市场上开 6 万月薪都招不到。

## Hooks 能做到的部分

用 `Stop` hook（agent 每次回合结束触发）可以自动执行 git 操作。配置示例：

```json
// .claude/settings.json
{
  "hooks": {
    "Stop": [
      {
        "type": "command",
        "command": "path/to/auto-commit.sh"
      }
    ]
  }
}
```

## 但有一个关键限制

Hooks 本质上跑的是 **shell 脚本**，它没有 LLM 推理能力。它能拿到：
- `git diff` 的输出（改了哪些文件的哪些行）
- 工具调用的输入输出

但它**无法理解改动的业务含义**，所以 commit message 只能是机械式的，比如：

```
自动提交: 改动 3 个文件
- src/foo.ts: +5 -2
- src/bar.ts: +12 -0
```

而**不是**你期望的"修复了 XX bug，原因是 YY，方案是 ZZ"这种有意义的描述。

## 两个可行方案

**方案 A：自动 commit + 机器式 message**
每次 turn 结束自动 commit，message 带 diff 摘要。简单粗暴，但 commit 历史可读性一般。

**方案 B：让 Claude 自己在完成时 commit（推荐）**
不用 hooks，直接在 CLAUDE.md 里加一条规则：

```markdown
## Git 规范
每次完成一个功能点或修复后，主动执行 git commit。
commit message 使用简体中文，描述清楚改了什么、为什么改。
```

这样每次 commit message 都是有语义的，而且 Claude 刚做完这些改动，对上下文最清楚。

## When Hooks Are Required (Not Memory)

If the user wants something to happen automatically in response to an EVENT, they need a **hook** configured in settings.json. Memory/preferences cannot trigger automated actions.

**These require hooks:**
- "Before compacting, ask me what to preserve" → PreCompact hook
- "After writing files, run prettier" → PostToolUse hook with Write|Edit matcher
- "When I run bash commands, log them" → PreToolUse hook with Bash matcher
- "Always run tests after code changes" → PostToolUse hook

**Hook events:** PreToolUse, PostToolUse, PreCompact, PostCompact, Stop, Notification, SessionStart

## CRITICAL: Read Before Write

**Always read the existing settings file before making changes.** Merge new settings with existing ones - never replace the entire file.

## CRITICAL: Use AskUserQuestion for Ambiguity

When the user's request is ambiguous, use AskUserQuestion to clarify:
- Which settings file to modify (user/project/local)
- Whether to add to existing arrays or replace them
- Specific values when multiple options exist

## Decision: Config Tool vs Direct Edit

**Use the Config tool** for these simple settings:
- `theme`, `editorMode`, `verbose`, `model`
- `language`, `alwaysThinkingEnabled`
- `permissions.defaultMode`

**Edit settings.json directly** for:
- Hooks (PreToolUse, PostToolUse, etc.)
- Complex permission rules (allow/deny arrays)
- Environment variables
- MCP server configuration
- Plugin configuration

## Workflow

1. **Clarify intent** - Ask if the request is ambiguous
2. **Read existing file** - Use Read tool on the target settings file
3. **Merge carefully** - Preserve existing settings, especially arrays
4. **Edit file** - Use Edit tool (if file doesn't exist, ask user to create it first)
5. **Confirm** - Tell user what was changed

# Markdown 导入技能

从本地 Markdown 文件创建或更新飞书云文档。**支持 Mermaid/PlantUML 图表转飞书画板、大表格自动拆分**。

> **CRITICAL：** 每次创建新文档后，**必须立即**执行以下两步：
> 1. 授予 `full_access` 权限：`feishu-cli perm add <document_id> --doc-type docx --member-type email --member-id user@example.com --perm full_access --notification`
> 2. 转移文档所有权：`feishu-cli perm transfer-owner <document_id> --doc-type docx --member-type email --member-id user@example.com --notification`
>
> 详见下方"执行流程 → 创建新文档"。

## 核心特性

1. **三阶段并发管道**：顺序创建块 → 并发处理图表/表格 → 失败回退
2. **Mermaid/PlantUML → 飞书画板**：`mermaid`/`plantuml`/`puml` 代码块自动转换为飞书画板
3. **图表故障容错**：语法错误自动降级为代码块展示，服务端错误自动重试（最多 10 次，1s 间隔）
4. **大表格自动拆分**：超过 9 行或 9 列的表格自动拆分为多个表格，行拆分保留表头，列拆分保留首列作为标识
5. **表格列宽自动计算**：根据内容智能计算列宽（中英文区分，最小 80px，最大 400px）
6. **API 限流处理**：自动重试，避免 429 错误
7. **并发控制**：图表和表格分别使用独立的 worker 池（默认图表 5、表格 3 并发）

## 核心概念

**Markdown 作为中间态**：本地文档与飞书云文档之间通过 Markdown 格式进行转换。

## 前置条件

- **feishu-cli**：如尚未安装，请前往 [riba2534/feishu-cli](https://github.com/riba2534/feishu-cli) 获取安装方式
- 已配置 App Token（`FEISHU_APP_ID` + `FEISHU_APP_SECRET`），无需 `auth login`
- Markdown 文件使用 UTF-8 编码

## 这种多Agent协作分析报告怎么做出来的

你看到的这类报告（多个Agent分别分析，最后合成一份完整文档），本质上是 **"MapReduce for LLMs"** 模式，业内已经有一套比较成熟的玩法了。

### 最经典的案例

**Hrishi（Southbridge AI CEO）的 "Claude Code: An Analysis"** 是目前公开的最完整案例，知乎有详细的中文复盘：[知乎链接](https://zhuanlan.zhihu.com/p/1930393662669447391)

他迭代了 **5 个版本** 才找到最佳方案：

| 版本 | 方案 | 结果 |
|------|------|------|
| V1 | 人当协调员，手动搬运上下文 | 能用但太累 |
| V2 | Agent之间直接通信 | 翻车，开始互相瞎编 |
| V3 | Opus 4 当超级Agent审阅+派活 | 2轮就爆上下文 |
| V4 | Opus 4 无状态派发（每次重写prompt） | 好一些但出现幻觉 |
| **V5** | **Opus 4 单一宽prompt + 子Agent保持状态 + 人工清理** | 最佳，跑了5轮有效迭代，还出现了**涌现行为**——Opus自己开始给问题标优先级 |

### 通用工作流（5步）

```
1. 分解 → 把主题拆成3-10个维度
2. 并行分发 → 每个维度分配一个子Agent分析
3. 收集+清洗 → 去掉幻觉、重复、无关内容
4. 交叉验证 → 超级Agent检查矛盾和遗漏，追问子Agent
5. 顺序合成 → 单个写作Agent一口气写完最终报告（关键：不能并行写）
```

### 现成工具/框架

| 工具 | 特点 |
|------|------|
| **Claude Code Subagents** | 最直接的方式，一个session里spawn多个子agent，每个有独立上下文，结果返回给父agent合成 |
| **LangChain Open Deep Research** | 开源深度研究agent，自动拆分+并行+合成 |
| **CrewAI** | 定义角色（Researcher/Analyst/Writer）和任务，按流程协作 |
| **Gastown**（Steve Yegge做的） | 多Agent编排器，"城市管理"模式，git原生持久化 |
| **33个预定义Subagent** | [GitHub](https://github.com/lst97/claude-code-sub-agents) 现成可用的子agent定义集合 |

### 实操建议

如果你想用 **Claude Code（也就是咱们HappyClaw这套）** 做类似的事，路径很清晰：

1. **用 Agent tool 的 `subagent_type="general-purpose"`** 并行启动多个子agent，每个负责一个分析维度
2. 子agent返回结果后，主agent负责交叉验证和合成
3. 最终报告用单个agent顺序写入，保证连贯性

- **别在agent之间做摘要传递**，会丢失关键上下文

- **人工清理环节不能省**，agent的幻觉和冗余必须人工过一遍

**HappyClaw**: Claude Code 的 SubAgent 开发有几种方式，从简到复杂：

### 1. 内置 Agent Tool（最常用）

在对话中直接用，不用写任何代码：

```
// 启动一个通用子Agent做研究
Agent(subagent_type="general-purpose", prompt="搜索xxx并总结")

// 启动代码审查Agent
Agent(subagent_type="code-reviewer", prompt="审查 src/index.ts")

// 启动网页研究Agent
Agent(subagent_type="web-researcher", prompt="调研xxx的最新进展")
```

子Agent有独立的上下文窗口，跑完结果自动返回给主Agent。

### 2. 自定义 SubAgent 定义文件

在 `.claude/agents/` 目录下放 YAML 文件，定义专属子Agent：

```yaml
# .claude/agents/security-reviewer.md
name: Security Reviewer
description: Review code for security vulnerabilities
model: sonnet  # 可选，默认跟随主Agent
prompt: |
  You are a security expert. Review the given code for:
  - OWASP Top 10 vulnerabilities
  - Injection attacks
  - Authentication/authorization flaws
  Output a severity-rated vulnerability report.
```

定义好后就能通过 `Agent(subagent_type="security-reviewer")` 调用。

### 3. Agent Teams（多会话协作）

多个 Claude Code 会话协同工作，适合长期独立任务：

```bash
# 会话1：团队领导
